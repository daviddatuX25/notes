T11 "How does RAG prevent hallucinations?"
A RAG chatbot searches a curated knowledge base of ISPSC policy documents first — if it finds relevant text, that becomes ground truth for the response; if no document matches above the confidence threshold, it states it does not have the information rather than guessing.
· Standard LLM → training data (stale/wrong). RAG → search KB first   · Match → ground truth. No match → "I don't have that information"   · Retrieval anchors every response to institutional documents
T12 "Why RAG vs fine-tuning?"
Fine-tuning is static and expensive — every policy change requires retraining; with RAG, updating a document in the vector database gives the chatbot the new information immediately, with zero retraining overhead.
· Fine-tuning = static + expensive (retrain per change)   · RAG = update vector DB text → immediate   · Real-time accuracy without training overhead
T17 "How is K defined? What features?"
K represents ISPSC's program categories — STEM, Business, Arts, Education — set by the institution's academic structure; features are sub-test scores from the admission exam, and the K-Means engine is swappable via the Model Adapter pattern.
· K = ISPSC program categories (STEM, Business, Arts, Education)   · Features: sub-test scores (Math, Logic, Reading Comp, subject sections)   · Model Adapter → K-Means swappable without system changes
T18 CRITICAL "AI making admission decisions — dangerous?"
K-Means is strictly descriptive, not prescriptive — it surfaces aptitude patterns for counselor review but does not admit or reject anyone; every placement requires manual counselor confirmation recorded in the audit log.
· Descriptive, not prescriptive — does NOT admit/reject   · Counselor = final human call   · 2 safeguards: manual confirmation + HMAC-SHA256 audit log
T24 "Dash stopped at design. Why will you succeed?"

· Dash: concept proven. Us: methodology advanced   · Traditional dev vs AI-augmented RAD   · Every component now has published implementation

Q&A — Objectives
"How do the four objectives justify the feature set?"
The SOs form a dependency chain — diagnose (SO1), specify requirements (SO2), build, then measure acceptance (SO3) and prove improvement (SO4); each feature traces to a gap found in SO1 and a requirement specified in SO2.
· Dependency chain: SO1→SO2→SO3→SO4   · Each feature traces to SO1 gap + SO2 requirement — evidence-driven
T67 "SO3 measures acceptance, SO4 measures improvement. Aren't they the same thing?"
No — SO3 asks 'is NEXIAM acceptable?' (absolute), SO4 asks 'is NEXIAM better than manual?' (relative). SO3 could pass while SO4 fails if the manual process is already well-regarded; SO4 could pass while SO3 is marginal if the baseline is low. They answer different questions and require different statistical treatment.
· SO3 = absolute acceptance (mean > 4.0 midpoint)   · SO4 = relative improvement (paired delta)   · Independently informative — one can pass while other fails
T27 CRITICAL "85% of developers use AI, 41% of code is AI-generated — where in your manuscript?"
The statistics are from Osmani's 2026 Google whitepaper cited in the Software Model section — the manuscript describes the paradigm shift qualitatively, the slide provides the quantitative anchor from the same source. Peng et al. (2023) carries the peer-reviewed productivity evidence in the manuscript body.
· Source: Osmani (2026) Google whitepaper — cited in Software Model   · Manuscript = qualitative; slide = quantitative from same source   · Peng (2023) = peer-reviewed weight in manuscript body
T68 "How do you know SO1 gaps are real and not just anecdotal complaints?"
Triangulation — every gap in SO1 is confirmed through two independent sources: structured field observation with pre-defined event categories measuring scoring time and posting method, and stakeholder confirmation where extracted themes are brought back to Guidance and Registrar for verification. Anecdotes become requirements only when confirmed.
· Triangulation: field observation + stakeholder confirmation   · Pre-defined event categories — measurable, not impressions   · "Is this accurate?" → only confirmed → requirements
T69 "SO4 uses paired t-test. But what if the data isn't normal?"
Shapiro-Wilk gates the decision — if normality holds, paired t-test; if it fails, Wilcoxon signed-rank. The data decides the statistical path, not us. Both tests are pre-specified before data collection to prevent post-hoc test shopping.
· Shapiro-Wilk → gates parametric vs non-parametric   · Wilcoxon = distribution-free fallback   · Pre-specified — no post-hoc test selection
T70 "Your objectives don't mention deployment. Is this just a prototype?"
SO3 and SO4 evaluate a system deployed on a Dokploy-managed VPS — not a localhost prototype. The evaluation uses the production deployment with real browser access, real RBAC enforcement, and real database operations. Dokploy gives containerized isolation, SSL, and health monitoring at capstone budget.
· Production VPS deployment, not localhost   · Dokploy → containerized, SSL, monitoring   · Evaluators access real running system

SLIDE 8 — Research Design & Methodology Overview Jaypee · ~1:15
Jaypee:
"Our methodology is a dependency chain. We classified the study as Descriptive-Developmental under Frey — diagnose first, build second."

(Layer 2) "The Descriptive Phase uses interviews and field observation to map the current process, then thematic analysis to extract verified gaps. Not every complaint becomes a feature — only gaps confirmed through triangulation graduate to requirements."

(Layer 3) "With requirements set, we construct NEXIAM via AI-Augmented RAD — four phases with a mandatory human review gate on every module. AI writes boilerplate; the team decides what ships."

Q&A — Research Design, Methodology Justification
T61 CRITICAL "Why not just Descriptive? Or just Developmental?"
Either alone truncates the study — descriptive-only maps without solving; developmental-only builds without requirements. Descriptive-Developmental is necessary because the outcome is a tested system, not a diagnosis or a prototype alone (Frey, 2022).
· Descriptive-only = maps, doesn't solve   · Developmental-only = builds without requirements   · Both = tested system
T62 "Your layers just restate the SOs. Redundant?"
SOs state what; the methodology explains why that order — you cannot build before diagnosis, and you cannot claim improvement without testing. A list of objectives does not enforce that constraint.
· SOs = what; Methodology = why that order + dependency constraint   · Enforces: no diagnosis → no build → no claim
T63 "Pre/post test = quasi-experimental. Why descriptive-developmental?"
Classified by primary activity — describe, develop, evaluate (Frey). The paired TAM is an evaluation tool within the developmental phase, not an experiment — no random assignment, no IV manipulation.
· Primary activity = describe → develop → evaluate   · Pre/post = evaluation tool, not experiment   · No random assignment → not quasi-experimental
T64 "Thematic analysis in a capstone timeline — realistic?"
Targeted dataset — 5–7 interviews across two offices, validated through field observation triangulation. Goal is operational requirement extraction, not theoretical saturation (Braun & Clarke, 2006).
· Targeted, not open-ended (5–7 interviews)   · Triangulation via field observation   · Goal = operational requirements, not theoretical saturation

Q&A — Statistics, Analysis Validity
T50 "Shapiro-Wilk low power at n=46?"
It detects gross departures reliably; for subtle violations, we report exact W and p-values transparently, report effect size (distribution-robust), and can default to Wilcoxon — which makes no distributional assumptions — as the primary test.
· Detects gross departures; may miss subtle   · 2 safeguards: report exact W + p, report effect size (robust)   · Option: default to Wilcoxon
T51 "Why effect size + p-value?"
A p-value tells you a difference exists; effect size tells you how big — n=46 can produce significant p for a trivial shift, so Cohen's d or rank-biserial correlation quantifies practical meaning, as required by APA 7th.
· p = difference exists? | d = how big?   · n=46 → significant p possible for trivial shift   · APA 7th standard: report both
T52 "Thematic analysis — how validate themes?"
Triangulation against field observation notes and stakeholder confirmation — extracted themes are brought back to Guidance and Registrar offices for verification; only confirmed themes become functional requirements in SO2.
· Triangulation: field observation + stakeholder confirmation   · "Is this accurate?" → Only confirmed → requirements
T53 "TAM mean > 4.0 = acceptance? Barely neutral."
4.0 is the scale midpoint — the standard TAM interpretive benchmark, not a bar we invented; we report exact means and standard deviations per construct per role transparently, and the paired test answers the complementary question: is it better, not just acceptable?
· 4.0 = scale midpoint (positive/negative boundary) — TAM standard   · Report exact means + SD per role   · Paired test: is it better?
T54 "Why only PU + PEOU hypotheses?"
PU and PEOU are the core formative constructs in Davis's TAM model — ATU and BIU are downstream mediators that logically follow if the core constructs show improvement; testing all four inflates the familywise error rate to approximately 19%.
· PU + PEOU = core constructs (Davis 1989). ATU + BIU = downstream mediators   · Core improvement → downstream logically follows   · Testing all 4 → familywise error α ≈ .19
